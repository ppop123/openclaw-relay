# OpenClaw Relay v0.5.0

`OpenClaw Relay v0.5.0` is now available.

This minor pre-1.0 release focuses on **durability hardening for the agent-only peer capability**, **stricter peer-method boundaries**, and **more reproducible docs and usage flow**.

## Highlights

- Fixes peer streaming so it no longer hangs when only `stream_end` arrives and no trailing `response` follows
- Fixes peer stream redial so it does not replay duplicate chunks after partial output was already delivered
- Fixes peer auto-accept so it no longer consumes non-`invite_request` signals and races with manual dial / connect flows
- Adds code-level **agent-only / operator-only** guards to all `relay.peer.*` gateway methods so browser-like human-facing clients cannot invoke them
- Coalesces concurrent dial attempts, adds a small redial backoff, and changes `relay.peer.request` semantics to “queued to relay” instead of pretending delivery is guaranteed
- Tightens Quick Start, README, source-first engineering guidance, and the soak script to remove misleading or non-reproducible behavior

## Official Release Scope

The official support surface of `v0.5.0` is unchanged:

- `relay/` — Go relay server
- `sdk/python/` — Python client SDK (Layers 0–2)
- `client/` — Web reference client
- `plugin/` — OpenClaw gateway plugin
- `protocol/` — protocol docs, error codes, and canonical fixtures

## Not Included in the Official Release Scope

The following remain outside the official release scope:

- `sdk/js/` — not yet implemented
- future protocol extensions beyond the currently documented surface

## What Changed in v0.5.0

- This is a minor pre-1.0 release: it does not expand the official protocol surface, but it materially improves the durability of the agent-only peer capability
- `relay.peer.*` now enforces the product boundary in code: agents may collaborate, but human-facing clients may not overreach
- README / Quick Start now make the pairing flow reproducible: `enable` first, then complete the browser connection while `pair --wait` is still open
- OpenClaw-facing engineering guidance is tighter as well: when behavior depends on OpenClaw runtime semantics, local source inspection is mandatory before implementation
- `v0.5.0` is now the recommended `v0.5.x` release going forward

## Notes

- `v1` still does **not** provide forward secrecy
- The browser gateway verification model is **user-supplied pinned public key**, not automatic TOFU
- The agent-only peer capability still preserves the product boundary: human-facing clients may talk only to their own OpenClaw and must not discover or contact other OpenClaw instances
