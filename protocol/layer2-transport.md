# Layer 2: Transport Protocol

The transport layer provides request/response semantics, streaming, and multiplexing over the encrypted channel. All messages at this layer are encrypted by Layer 1 before transmission.

## Message Envelope

Every Layer 2 message is a JSON object with a common header:

```json
{
  "id": "<request_id>",
  "type": "<message_type>",
  ...type-specific fields
}
```

- `id`: A unique identifier for correlating requests and responses. Generated by the sender. Suggested format: `msg_<random_hex_8>`.
- `type`: One of the defined message types below.

## Message Types

### REQUEST

Client sends a request to the gateway:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "request",
  "method": "<method_name>",
  "params": { ... }
}
```

- `method`: The application-layer method to invoke (defined in Layer 3).
- `params`: Method-specific parameters.

### RESPONSE

Gateway responds to a request:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "response",
  "result": { ... }
}
```

Or, if an error occurred:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "response",
  "error": {
    "code": "<error_code>",
    "message": "<description>"
  }
}
```

The `id` matches the original REQUEST.

### STREAM_START

Gateway signals the start of a streaming response:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "stream_start",
  "method": "<method_name>"
}
```

### STREAM_CHUNK

Gateway sends a chunk of streaming data:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "stream_chunk",
  "seq": 1,
  "data": { ... }
}
```

- `seq`: Monotonically increasing sequence number within this stream, starting at 1.
- `data`: Chunk content (format defined by the method).

### STREAM_END

Gateway signals the end of streamed chunks:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "stream_end",
  "seq": 42
}
```

`stream_end` closes the chunk stream only. For Layer 3 methods that expose final metadata (for example `chat.send` returning the resolved `session_id` or token usage), the gateway MAY send a trailing `response` message with the same `id` after `stream_end`.

### CANCEL

Client cancels an in-flight request or stream:

```json
{
  "id": "msg_a1b2c3d4",
  "type": "cancel"
}
```

The gateway SHOULD stop processing and may send a final RESPONSE with error code `"cancelled"`.

### NOTIFY

Either side sends a one-way notification (no response expected):

```json
{
  "id": "msg_e5f6g7h8",
  "type": "notify",
  "event": "<event_name>",
  "data": { ... }
}
```

Used for: agent status changes, cron completion notifications, connection state updates.

## Multiplexing

Multiple requests can be in-flight simultaneously. Each has a unique `id`. Responses and stream chunks are correlated by `id`. The transport layer does not enforce ordering between different request IDs.

Example of interleaved streams:

```
← stream_chunk  id=msg_001 seq=1  data="Hello "
← stream_chunk  id=msg_002 seq=1  data="The weather "
← stream_chunk  id=msg_001 seq=2  data="world!"
← stream_end    id=msg_001 seq=3
← stream_chunk  id=msg_002 seq=2  data="is sunny."
← stream_end    id=msg_002 seq=3
```

## Timeouts

- REQUEST timeout: 120 seconds (configurable by the client). If no RESPONSE or STREAM_START is received, the client should consider the request failed.
- STREAM idle timeout: 60 seconds. If no STREAM_CHUNK is received for 60 seconds during an active stream, the client may send CANCEL.
- These are application-level timeouts, independent of WebSocket/relay keepalive.

> **Note:** These timeouts are suggestions. Clients MAY adjust them based on expected response latency (e.g., AI inference can take longer than 120 seconds for complex tasks). The protocol does not enforce timeouts at the relay level.

## Flow Control

The protocol does not define explicit flow control. In practice:
- The relay enforces a per-channel message rate limit (Layer 0).
- The WebSocket transport provides TCP-level flow control.
- Clients should limit concurrent in-flight requests to a reasonable number (suggested: 16).

## Retry Semantics

The protocol does not guarantee exactly-once delivery. Implementations should be aware of the following:

- **Requests are not idempotent by default.** If a client does not receive a RESPONSE or STREAM_START within the timeout, it cannot safely assume the request was not processed. Methods that are safe to retry (e.g., read-only queries like `agents.list`) should be documented as idempotent in Layer 3.
- **Clients SHOULD NOT automatically retry `chat.send`** or other state-changing methods without user confirmation, as the gateway may have already processed the request.
- **Stream resumption is not supported in v1.** If a stream is interrupted (disconnect, timeout), the client must send a new REQUEST. The gateway may return a cached result or re-execute, depending on the method.
- **The transport layer does not deduplicate.** If the same `id` is reused across reconnections, behavior is undefined. Clients MUST generate a new `id` for each request.

## Error Codes

| Code | Description |
|------|-------------|
| `method_not_found` | Unknown method name |
| `invalid_params` | Invalid or missing parameters |
| `internal_error` | Gateway encountered an unexpected error |
| `cancelled` | Request was cancelled by the client |
| `timeout` | Gateway timed out processing the request |
| `not_authenticated` | Client has not completed Layer 1 handshake |
| `permission_denied` | Client is not authorized for this method |
